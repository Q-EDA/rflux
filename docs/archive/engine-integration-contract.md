# rflux 引擎集成契约（Phase E1 草案）

## 1. 目的

本文件定义 `rflux` 作为可嵌入能力层（engine）时，对桌面端前端、外部编排层和服务适配层暴露的最小稳定契约。

本文件不定义 REST、WebSocket、SSE 或常驻服务协议；这些属于集成方适配层职责。

## 2. 边界

本契约覆盖：

- 引擎调用输入/输出的最小语义边界。
- 长任务的结构化进度事件模型。
- 执行控制语义（取消、超时、静默默认行为）。
- 结构化错误与诊断包的最小字段集。

本契约不覆盖：

- 用户认证、租户隔离、队列调度、审计计费。
- 远程服务部署拓扑。
- 服务端会话生命周期与网络重连协议。

## 3. 契约分层

### 3.1 Engine Contract（本仓库维护）

- Rust crate、Python API、CLI 的公共能力面。
- 进度事件 schema。
- 结构化错误字段与错误码语义。
- 诊断包最小字段集。
- CLI First 默认行为与兼容要求。

### 3.2 Service Adapter Contract（集成方维护）

- 如何把引擎事件映射到 REST/WebSocket/SSE/IPC。
- 如何实现队列、鉴权、资源配额与审计。
- 如何把引擎取消语义映射为前端按钮和会话控制。

## 4. 默认行为

### 4.1 CLI First 默认规则

- 默认 CLI 路径必须保持单机、单命令、无常驻服务依赖。
- 默认 CLI 输出必须保持脚本友好；新增结构化进度默认不得污染 `stdout`。
- 进度、事件、诊断增强能力必须显式开启，或经明确的文件/回调出口输出。

### 4.2 嵌入式默认规则

- Rust/Python 嵌入调用可选择传入进度回调、事件接收器或取消句柄。
- 若未提供这些增强入口，引擎必须退化为与当前 CLI 一致的静默执行语义。

## 5. 长任务进度事件模型

进度事件面向“引擎正在做什么”，而不是面向某种网络协议。

### 5.1 最小字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `schema_version` | 是 | 当前事件 schema 版本 |
| `sequence` | 是 | 单次执行内严格单调递增序号 |
| `operation` | 是 | 当前操作类型，如 `simulate-file`、`compile-layout` |
| `stage` | 是 | 当前阶段，如 `parse`、`flatten`、`solve`、`postprocess` |
| `status` | 是 | `started` / `running` / `completed` / `failed` / `canceled` |
| `message` | 否 | 面向人类的简短说明 |
| `percent` | 否 | 0-100 的近似总进度 |
| `stage_current` | 否 | 当前阶段已处理计数 |
| `stage_total` | 否 | 当前阶段总计数 |
| `eta_ms` | 否 | 估计剩余毫秒数 |
| `timestamp` | 否 | 事件生成时间戳 |

### 5.2 约束

- `sequence` 必须可用于事件去重和断点续接。
- `percent` 可近似，不要求严格线性。
- 不得把仅供调试的瞬时内部状态强制暴露为稳定字段。

### 5.3 阶段命名原则

- 阶段名应优先体现稳定语义边界，而非内部实现细节。
- 可接受先从粗粒度阶段开始，后续仅做向后兼容扩展。

## 6. 执行控制契约

### 6.1 取消

- 引擎应支持可观测取消语义。
- 长循环和阶段边界必须有取消检查点。
- 取消应进入可识别状态，而非表现为通用失败。

### 6.2 超时

- 超时属于显式控制输入，而不是隐式环境副作用。
- 超时触发后应返回可区分的错误语义或终止状态。

### 6.3 默认静默

- 未启用事件输出时，不得因引擎增强而改变当前 CLI 主结果、退出码和默认输出。

## 7. 结构化错误契约

### 7.1 最小字段

| 字段 | 必填 | 说明 |
|---|---|---|
| `code` | 是 | 稳定错误码，如 `RFLOW-*` |
| `message` | 是 | 面向用户的简短错误说明 |
| `detail` | 否 | 更具体的上下文 |
| `next_action` | 否 | 推荐下一步动作 |
| `retryable` | 否 | 是否适合重试 |
| `stage` | 否 | 错误发生阶段 |

### 7.2 约束

- 错误码优先稳定，`message` 可做措辞优化。
- 前端集成增强不得把原有稳定错误码退化为泛化内部错误。

## 8. 诊断包契约

诊断包仍以当前 `collect-diagnostics` / `run-with-diagnostics` 产物为基线。

### 8.1 最小内容

- `manifest.json`
- `events.jsonl`
- 输入文件快照（如 `inputs/`）
- 业务报告或生成报告（如 `reports/`）
- 执行摘要、错误码、最小环境摘要

### 8.2 目标

- 支持复现单次失败。
- 支持前端或外部服务层保留结构化证据。
- 不要求一次性覆盖完整商业支持包的全部字段。

## 9. 兼容等级

当前建议：

- 进度事件 schema：`limited`
- 执行控制契约：`limited`
- 结构化错误字段：`stable`（在已存在错误码范围内）
- 诊断包最小字段集：`limited`

当字段从实验转为正式时，应同步更新：

- `docs/interface-inventory.md`
- `docs/release-policy.md`
- `docs/alignment-scorecard.md`

## 10. 发布要求

若候选发布触及本契约中的公共字段或默认行为，至少应附带：

1. 契约变更摘要。
2. 对应 CI / 测试锚点。
3. 是否影响 CLI First 默认语义的结论。
4. 若存在破坏性影响，必须给出版本化或豁免说明。

## 11. 当前执行建议

1. 先在长任务最多的命令面建立粗粒度阶段事件，不追求一步到位的细粒度百分比。
2. 先把进度事件设计成引擎内统一结构，再由集成方选择映射为 IPC 或服务协议。
3. 先守住 CLI 默认静默和脚本兼容，再逐步扩展增强能力。
