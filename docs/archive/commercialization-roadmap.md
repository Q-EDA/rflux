# rflux 商业化成熟工具开发路线图

## 1. 文档目的

本路线图的目标不是给出一个“看起来完整”的愿望清单，而是给出一套足以把 `rflux` 从当前研究型原型推进到商业化成熟工具的执行方案。

这里的“商业化成熟工具”指的是：

- 可以由外部付费用户在真实项目中使用，而不是仅供内部研究验证。
- 团队能够对功能边界、质量、兼容性、性能、版本升级和缺陷修复作出明确承诺。
- 输出结果在已声明边界内可重复、可验证、可追责。
- 工具具备稳定的输入输出契约、发布流程、支持流程和质量门禁。

这份文档按以下原则组织：

- 每个阶段都定义可验证的退出条件，而不是模糊的“能力增强”。
- 每条主线都要落到代码、测试、文档、CI、发布和支持机制。
- 不把“补功能”误当成“商业成熟”。
- 明确资源约束。如果资源不满足，本路线图无法按期达成目标。

## 2. 当前状态判断

基于仓库当前代码、文档和 CI 现状，`rflux` 的定位仍然是“工程化程度较高的研究型原型”，而不是商业成熟工具。当前状态可以概括为：

- 核心 Rust workspace 架构已经成型，模块边界较清晰。
- 综合、布局、布线、时序、验证、仿真、CLI、Python facade 都已有可执行实现。
- 仓库已有较高密度的 Rust 与 Python 测试，说明项目不是纸面设计。
- 但物理实现、时序、仿真、前端格式、PDK、结果可信度、发布与支持体系仍然处于 prototype / partial / bootstrap 阶段。

当前与商业成熟工具之间的核心差距，不在“再多几个 API”，而在以下六类能力仍不成立：

1. 结果可信度仍未建立到客户可依赖的程度。
2. 输入输出契约与生态兼容性仍不够稳定。
3. PDK / library / benchmark / signoff 体系仍不完整。
4. 性能、扩展性和可观测性尚未进入持续经营状态。
5. 发布、升级、回退、支持、兼容性承诺尚未产品化。
6. 缺少商业交付所需的文档、SLA、缺陷分级和客户成功流程。

### 2.1 对标复核结论（2026-05-28）

基于当前仓库代码与 CI（`checks`、`core-smoke-windows`、`waveform-compare-gate`）复核后，当前“对标软件功能对齐”应表述为：

- 与 Yosys 家族重叠的 IR/综合/SAT/等价/CLI 核心链路：`高对齐（局部强对齐）`。
- 与 Quaigh 对齐的布尔优化行为与夹具回归：`高对齐（仍有边界）`。
- 与 JoSIM 的仿真语义与工程成熟度：`部分对齐（partial），非同级替代`。

因此，当前可对外表述为“具备可执行且工程化程度较高的 SFQ EDA 原型工具链”，但不应表述为“已与成熟对标软件全面功能对齐”。

## 3. 商业化目标定义

只有当以下八个维度全部达标时，才可宣称 `rflux` 达到商业化成熟工具水平。

### 3.1 功能完整性

- 已公开承诺的主要流程在目标边界内可用：导入、综合、P&R、STA、验证、仿真、报告导出。
- 每项流程都有清晰的支持范围、限制条件和错误语义。
- 不依赖“用户知道哪些输入别碰”的隐性规则。

### 3.2 结果可信度

- 与外部基准工具或权威数据集建立长期对照。
- 对所有关键报告项建立回归阈值和漂移告警。
- 已知不支持的行为必须显式报错，不能静默退化。

### 3.3 兼容性与互操作

- 支持商业用户实际会交付的输入格式和数据路径。
- PDK、library、benchmark、CLI、Python API 都有版本化契约。
- 新版本不会无提示破坏现有用户流程。

### 3.4 性能与规模

- 对目标电路规模有公开、经过验证的容量边界。
- 对关键场景有基准测试和性能回归监控。
- 对超界输入有合理的失败模式，而不是卡死或不可解释的错误。

### 3.5 质量体系

- 单元测试、集成测试、回归测试、基准测试、模糊测试、兼容性测试全部进入 CI。
- 每次发布都有可追溯构建产物、测试记录和版本说明。
- 高优先级缺陷有明确的响应和修复流程。

### 3.6 产品化与发布

- 提供稳定 CLI、Python 包和版本化文档。
- 有 alpha / beta / GA 的发布准入标准。
- 有安装、升级、回滚、许可证和依赖策略。

### 3.7 运维与支持

- 有可复现问题收集模板、诊断包、日志级别和最小排障指引。
- 有支持矩阵：支持平台、Python 版本、Rust toolchain、外部依赖、PDK 版本。
- 有缺陷分级和 SLA。

### 3.8 组织能力

- 至少有一套稳定的职责分工覆盖内核、验证、PDK、QA、发布与文档。
- 关键知识不依赖单人记忆。
- 所有核心流程都可由第二人复现和维护。

## 4. 不可回避的现实约束

如果要真的把 `rflux` 做到商业化成熟，以下前提必须被接受。

### 4.1 时间预期

以当前项目基础估计，达到可以对外商业交付的程度，合理周期是 **18 到 30 个月**，不是几个迭代就能完成。

### 4.2 人员配置

建议最少配置如下：

- 2 名核心 Rust 算法 / 内核工程师。
- 1 名验证与质量工程师。
- 1 名 PDK / 工艺建模工程师。
- 1 名 Python / CLI / 产品化工程师。
- 0.5 到 1 名技术写作 / 开发者体验负责人。

如果只有 1 到 2 名兼职开发者，本路线图只能按顺序推进，无法在合理时间内达到商业成熟目标。

### 4.3 外部依赖

必须提前明确：

- 是否以 JoSIM / ColdFlux / qPALACE 等工具作为对照基准。
- 是否可以取得真实 PDK、库模型、 benchmark 和客户样例。
- 是否允许引入商业许可组件或仅限纯开源路径。

如果这些外部条件拿不到，最终目标只能是“高质量研究工具”，不是可验证的商业成熟工具。

## 5. 总体推进策略

路线图分为六个阶段，按顺序推进。原则上，前一阶段未达到退出条件，不进入下一阶段的承诺性交付。

| 阶段 | 名称 | 目标 | 预计周期 |
|------|------|------|----------|
| 0 | 基线冻结与商业化重构准备 | 建立统一目标、边界、质量门和项目治理 | 4-6 周 |
| 1 | 内核可信度与契约收敛 | 让现有原型在声明边界内稳定、可测、可解释 | 3-4 个月 |
| 2 | 前端、PDK 与生态接入 | 解决真实用户无法顺畅导入/建模/集成的问题 | 3-5 个月 |
| 3 | 结果可信度与 signoff 基础 | 建立对照、相关性和可追责的验证体系 | 4-6 个月 |
| 4 | 性能、规模与产品化交付 | 让工具能在目标规模上稳定运行并可发布 | 3-4 个月 |
| 5 | Beta 到 GA 商业发布 | 完成支持体系、文档体系、版本策略与首批客户闭环 | 3-5 个月 |

## 6. 阶段 0：基线冻结与商业化重构准备

### 6.1 阶段目标

- 把“要做成什么产品”定义清楚。
- 把“当前支持什么，不支持什么”定义清楚。
- 把“哪些指标代表成熟”定义清楚。

### 6.2 必须交付的产物

1. 产品范围说明书。
2. 支持矩阵 v0。
3. 能力边界说明书。
4. 缺陷分级与发布准入标准 v0。
5. 基准测试与对照验证方案。
6. 架构责任归属表。

### 6.3 具体任务

#### 6.3.1 冻结首个商业化目标边界

必须先明确第一代商业版本不做什么。建议只承诺一个窄而深的范围，例如：

- 输入前端先聚焦 IR JSON + 受限 SPICE/JoSIM deck + 一个可控的 HDL 子集。
- PDK 先支持 1 到 2 套可验证工艺库。
- 物理实现先限定特定规模、特定器件族和特定报告类型。
- 仿真先限定已验证子集，不把“未来要支持”写进首版对外承诺。

#### 6.3.2 建立商业化 KPI 仪表盘

从一开始就跟踪以下指标：

- 测试通过率。
- 崩溃率。
- 回归引入率。
- CLI/Python API 兼容性破坏次数。
- benchmark 性能中位数与 P95。
- 与外部参考工具的结果偏差。
- 已知限制项数量。
- 高优先级缺陷未关闭天数。

#### 6.3.3 统一错误分类

所有 Rust、CLI、Python 输出错误要统一分层：

- 用户输入错误。
- PDK / library 配置错误。
- 不支持的语法或能力边界。
- 内部算法失败。
- 资源不足或超出规模上限。
- 外部工具调用失败。

错误必须带机器可解析代码、稳定文本摘要和排障建议。

#### 6.3.4 建立发布治理

定义：

- `main` 是否允许直接提交。
- 每个 PR 的最小准入门。
- alpha / beta / GA 的出包条件。
- 紧急补丁发布流程。

### 6.4 退出条件

只有同时满足以下条件，阶段 0 才算完成：

- 已明确首个商业版本的支持边界。
- 已冻结 KPI 与质量门。
- 已建立 issue / epic / milestone 模板。
- 已明确阶段 1-5 的责任人和里程碑。

## 7. 阶段 1：内核可信度与契约收敛

### 7.1 阶段目标

让现有原型能力在声明边界内“稳定工作”，而不是继续快速扩张边界。

### 7.2 主攻方向

1. API 稳定性。
2. 错误语义稳定性。
3. 核心结果一致性。
4. 最小支持边界下的可靠回归。

### 7.3 具体任务

#### 7.3.1 冻结并版本化对外接口

要版本化的接口包括：

- CLI 子命令、参数、输出 JSON schema。
- Python facade 的公共类、函数、异常、字段。
- IR JSON / PDK JSON / report JSON schema。

要求：

- 每个 schema 都有显式版本号。
- 非向后兼容修改必须 bump major 或显式加迁移器。
- 引入 JSON schema 校验测试。

#### 7.3.2 收敛 placeholder / implicit behavior

当前所有占位 API、自动回退和隐式默认值都要梳理。

必须处理：

- placeholder API 改成真实实现，或者显式标记为实验接口。
- 所有 silent fallback 改为显式模式选择或显式警告。
- 默认 PDK / default flow config 的行为必须写入文档并进入测试。

#### 7.3.3 核心流程建立黄金回归

对以下流程建立 golden regression：

- `compile_netlist`
- `compile_layout`
- `analyze_timing`
- `verify_layout`
- `simulate_text`
- `simulate_file`
- `check-equivalence`
- `solve-dimacs`

每个流程至少覆盖：

- 正常输入。
- 边界输入。
- 非法输入。
- 不支持输入。
- 升级兼容输入。

#### 7.3.4 引入模糊测试和属性测试

至少覆盖：

- IR 读写 round-trip。
- DIMACS 解析与导出。
- SPICE 子集解析。
- 关键算法的不变量，例如 splitter / balance / route legality / timing monotonicity。

#### 7.3.5 建立 deterministic mode

商业工具必须支持可复现。

要求：

- 所有受随机性影响的路径都支持固定 seed。
- 同一输入、同一版本、同一配置在支持平台上结果可重复。
- 非确定行为必须可开关。

### 7.4 质量门

- 所有公共 API 都有兼容性测试。
- 所有公共报告都有 schema 测试。
- 所有错误码都有文档和最小回归。
- 新增能力必须先写失败测试，再写实现。

当前推进注记：

- Python facade 的高风险 silent fallback 正在按切片清零，`compile*`、`analyze_timing*`、`verify_layout` 已改为缺少已编译扩展时显式失败。
- CLI 顶层 JSON、IR JSON、PDK JSON 已开始版本化；其中 IR/PDK reader 暂时保留 legacy 裸 JSON 读取兼容，并已为 schema 版本不兼容和 envelope 非法建立最小回归。
- CLI 已开始在 `rflux-io` 驱动的输入 / schema 错误上输出 `error[RFLOW-...]`，说明错误码体系正从文档约定进入真实对外接口。

### 7.5 退出条件

- 对外接口全部版本化。
- 核心流程 golden regression 完整建立。
- 模糊测试进入 CI 夜跑。
- placeholder / silent fallback 基本清零或被标为实验能力。
- 能提供一份明确的“支持与限制”文档给外部试用用户。

## 8. 阶段 2：前端、PDK 与生态接入

### 8.1 阶段目标

解决“工具本身能跑，但真实用户接不进来”的问题。

### 8.2 主攻方向

1. 输入格式与数据导入。
2. PDK / library 建模与版本管理。
3. 与外部工具链的稳定接口。

### 8.3 具体任务

#### 8.3.1 输入前端商业化收敛

必须决定首版商业化支持哪些输入路径，并把每条路径做深：

- IR JSON：做成最稳的基线入口。
- SPICE / JoSIM deck：继续扩展，但只承诺已验证子集。
- HDL frontend：如果不能在首版做强，就不要对外承诺通用 Verilog 支持。

要求：

- 为每条支持路径提供语法支持表。
- 为每条不支持路径提供明确错误和迁移建议。

当前推进注记：

- IR JSON / PDK JSON 官方 writer 已统一写出 `schema_version + kind + payload` 包装对象。
- 读取路径仍接受历史裸 JSON，作为过渡兼容窗口；移除该窗口前必须先补迁移路径、兼容回归和 release notes 说明。
- CLI 已提供 `rflux lint-input --kind ir|pdk --input <path>` 作为输入预检查入口，可显式识别当前文件走的是版本化 envelope 还是 legacy 裸 JSON 兼容路径。

#### 8.3.2 PDK 体系产品化

商业化前必须建立正规的 PDK / library 生命周期：

- PDK schema 版本化。
- library characterization 流程固化。
- PDK 验证套件。
- PDK 变更影响分析。
- 每套 PDK 的支持矩阵与已知限制。

必须交付：

- `pdk validate` 类工具或等价 API。
- PDK 示例仓库或模板。
- PDK onboarding 文档。

#### 8.3.3 benchmark 与客户样例资产化

建立三层样例资产：

- 公共 benchmark。
- 工具对照 benchmark。
- 客户代表性样例。

每次改动必须能回答：

- 对 benchmark 的 QoR 影响是什么。
- 对客户样例的结果偏差是什么。
- 是否引入新的不支持语法或性能退化。

#### 8.3.4 Python 与 CLI 交付体验统一

要做到：

- CLI 与 Python 结果模型一致。
- 关键配置项命名一致。
- 文档中的示例可直接运行。
- 安装失败、依赖缺失、外部工具缺失时给出清晰诊断。

### 8.4 退出条件

- 至少 1 条输入路径达到商业级稳定，另 1-2 条输入路径达到受限支持。
- 至少 1 套真实 PDK / library 完成产品化接入。
- benchmark、样例、PDK 验证都进入 CI。
- CLI / Python / 文档三者语义对齐。

## 9. 阶段 3：结果可信度与 signoff 基础

### 9.1 阶段目标

这是最关键阶段。没有结果可信度，就没有商业化。

### 9.2 主攻方向

1. 与外部工具或权威数据建立对照。
2. 建立 QoR 和准确性阈值。
3. 建立 signoff 边界声明。

### 9.3 具体任务

#### 9.3.1 建立对照矩阵

按模块建立参考对照：

- 综合与逻辑等价：对照 Yosys / 内部基准结果。
- 物理实现：对照现有 SFQ 流程、手工布局或可信参考数据。
- STA：对照外部脚本、手工计算、历史签核样例。
- 仿真：对照 JoSIM 或权威 transient 结果。

每项对照必须定义：

- 使用什么输入集。
- 比较哪些指标。
- 允许的偏差范围。
- 偏差超限时谁负责分析。

#### 9.3.2 建立相关性回归

必须把“结果看起来差不多”升级成正式相关性工程：

- timing correlation dashboard。
- waveform correlation dashboard。
- route length / congestion / detour correlation。
- equivalence / SAT 结果一致性。

输出要求：

- 平均误差。
- P95 / worst-case 误差。
- 误差分布。
- 误差来源分类。

#### 9.3.3 建立 signoff 边界说明书

必须明确写出：

- 哪些报告可用于最终决策。
- 哪些报告仅用于 early exploration。
- 哪些模型仍属 heuristic。
- 哪些输入规模或器件类型不在当前签核范围。

任何不能被证明稳定的报告，都不得在商业合同或对外宣传中表述为 signoff 级能力。

#### 9.3.4 缺陷复盘机制

对所有高优先级偏差和错误，必须做 RCA：

- 根因是模型错误、实现错误、配置错误还是边界误用。
- 是否需要新增 benchmark。
- 是否需要新增 schema 约束。
- 是否需要限制功能承诺。

### 9.4 退出条件

- 已建立正式外部对照矩阵。
- 关键报告项均有误差阈值和长期趋势图。
- 首个商业范围内的能力可以写出可信边界声明。
- 高优先级偏差处理形成闭环流程。

## 10. 阶段 4：性能、规模与产品化交付

### 10.1 阶段目标

让工具在目标客户规模上“用得起来、跑得完、排得动问题、发得出去”。

### 10.2 主攻方向

1. 性能与容量。
2. 可观测性与诊断。
3. 安装、发布、升级、回滚。

### 10.3 具体任务

#### 10.3.1 建立性能基线

为各模块建立性能基线：

- 综合时长。
- P&R 时长。
- STA 时长。
- 仿真时长。
- 内存占用。
- 大规模样例成功率。

CI 中至少保留：

- 快速 smoke benchmark。
- 每日完整 benchmark。
- 每周历史趋势报表。

#### 10.3.2 可观测性产品化

商业工具必须让用户能定位问题。必须提供：

- 统一日志级别。
- 结构化日志。
- 诊断包导出。
- 配置回显。
- 运行摘要。
- 性能 profile 开关。

当前推进注记：

- CLI 已新增 `collect-diagnostics` 最小诊断包导出能力，当前会生成 `manifest.json`、记录工作目录、最小环境摘要与 `RFLOW_*` / `JOSIM_*` 变量名快照，并收集问题复现用到的输入文件副本和输入契约快照；同时已补最小 `configuration` / `summary` 区块，用于回显路径与仿真配置，并汇总 legacy 兼容输入、契约检查失败项以及已存在 JSON report 的 kind/schema 摘要。诊断包目录还会产出最小结构化日志 `events.jsonl`，记录采集开始、输入复制、report 归档和 manifest 准备等事件，作为支持侧最小可归档上下文。
- CLI 已补统一执行入口 `run-with-diagnostics`，当前已接通 `simulate-file`、`verify-layout`、`compile-layout`、`analyze-timing`、`compile-netlist`、`solve-dimacs`、`check-equivalence` 和 `lint-input`：执行真实命令时会直接产出 bundle，把执行开始/完成/失败事件写入 `events.jsonl`，并在 manifest 中记录最终 report、错误码以及最小 stdout/stderr 摘要。
- 当前诊断包已覆盖平台信息、CLI 版本、调用参数和最小环境摘要，并在 `simulate-file` / `verify-layout` / `compile-layout` / `analyze-timing` / `compile-netlist` / `solve-dimacs` / `check-equivalence` / `lint-input` 路径下具备最小运行摘要和执行结果归档；flow-oriented 路径的输入捕获、`command_started` 事件和公共 configuration 骨架也已在 CLI 内部完成共享化，说明这一轮“统一 diagnostics 执行骨架”的收敛工作已经基本收口。剩余 observability 缺口主要变为更多真实命令接入、profile 开关、完整配置快照和实时日志归档，而不是继续清理现有八条路径的重复实现。

下一阶段入口：

- 将错误码统一从 `rflux-io` 输入 / schema 错误继续推进到 `FLOW` / `VERIFY` / `SIM` 路径。
- 保证 diagnostics bundle 与普通 CLI stderr 使用同一套稳定错误码与建议动作。

#### 10.3.3 安装与分发体系

必须明确支持平台和分发方式，例如：

- Windows / Linux 版本矩阵。
- Python wheel 分发。
- Rust CLI 二进制分发。
- 可选的源码构建路径。

每次发布都必须提供：

- 发布说明。
- 变更影响说明。
- 升级步骤。
- 回滚方案。

#### 10.3.4 安全与合规

商业交付前必须补上：

- 依赖清单与许可证清单。
- 第三方组件风险审查。
- SBOM。
- 基础安全扫描。
- 外部命令调用安全边界。

当前推进注记：

- 仓库已补 `docs/security-compliance.md`，把当前许可证基线、Rust 依赖/许可证清单、基础安全扫描和外部命令边界先固定成文档化基线。
- CI 已提供可手动触发的 `security-compliance` 基线作业，当前产出 Rust 依赖清单、Rust 许可证清单、`cargo audit` 扫描结果、Python 依赖清单及 Python 依赖审查输入。
- 仿真路径中的 `external_command` 已补最小 allowlist、最小路径信任规则、最小环境隔离、最小副作用文件约束和策略文件，当前只允许 `josim` / `josim-cli` 及其受限 wrapper 后缀，并会移除 `RFLOW_*` / `JOSIM_*` 子进程环境变量，同时把单次运行输入收进独立 `rflux-ext-*` 临时目录；更完整的 sandbox、签名校验和环境/文件策略仍未完成。
- Python 许可证清单已补自动化导出，当前通过 `uv.lock` 中的 wheel 元数据生成机器可读 inventory，优先读取本地 `uv` 缓存，并显式列出缺失项与抓取失败项；这些项仍需要例外处理。
- 第三方组件风险审查清单、初始台账和补审模板已补，但正式的例外审批流程和长期跟踪机制仍未完成。
- 正式 SBOM 仍属于未完成项，暂不能表述为已产品化。

### 10.4 退出条件

- 已声明的目标规模上具备稳定性能与容量。
- 出现问题时可以由日志和诊断包复现。
- 已有可重复的安装、升级、回滚流程。
- 已完成基础安全与许可证合规检查。

## 11. 阶段 5：Beta 到 GA 商业发布

### 11.1 阶段目标

把“工具可用”升级成“产品可交付、团队可承诺”。

### 11.2 发布分级

#### Alpha

适用对象：内部团队和深度共研用户。

要求：

- 功能边界明确。
- 核心流程稳定。
- 已知限制公开。
- 日志与诊断可用。

#### Beta

适用对象：受控外部试点用户。

要求：

- 对外文档完整。
- PDK onboarding 稳定。
- 结果对照达标。
- 升级兼容策略明确。
- 高优先级问题修复周期可承诺。

#### GA

适用对象：正式商业用户。

要求：

- 关键能力达到正式支持标准。
- 兼容矩阵、SLA、支持流程、发布策略全部生效。
- 首批客户试点闭环完成。
- 重大缺陷率达到预设阈值以下。

### 11.3 必须交付的产品化资产

1. 用户手册。
2. 运维与排障手册。
3. API 参考文档。
4. PDK 接入文档。
5. 发布说明模板。
6. 缺陷分级与响应手册。
7. 客户升级指南。
8. 已知限制清单。
9. 兼容性矩阵。
10. 支持申请模板。

### 11.4 退出条件

- 至少完成 2 轮 Beta 试点。
- 试点问题全部归因并处理。
- GA 版本的支持材料齐备。
- 团队对支持范围、响应方式和升级影响有一致承诺。

## 12. 必须并行推进的七条工作流

阶段推进是纵向切分，但执行上必须并行跑七条工作流，缺一不可。

### 12.1 内核算法工作流

负责：综合、P&R、STA、仿真、验证核心实现。

要求：

- 新能力必须伴随 benchmark、错误语义和回归用例。
- 不允许只交实现，不交验证和文档。

### 12.2 输入输出与 schema 工作流

负责：IR、PDK、报告、CLI JSON、Python model 的 schema 契约。

要求：

- 每次 schema 改动都要经过兼容性评审。

### 12.3 PDK / library 工作流

负责：工艺建模、characterization、验证套件、版本控制。

要求：

- 每个 PDK 变更都必须知道它影响哪些 benchmark 和报告。

### 12.4 QA 与基准工作流

负责：CI、nightly、benchmark、fuzz、相关性验证、回归门禁。

要求：

- 质量门可以阻断发布。

### 12.5 产品与发布工作流

负责：安装、打包、版本说明、升级、回滚、兼容策略。

要求：

- 版本发布不能依赖手工知识。

### 12.6 文档与开发者体验工作流

负责：用户手册、API 文档、错误码文档、示例、教程。

要求：

- 文档中的所有命令都必须被 CI smoke 覆盖。
- 文档里进入支持或发布门禁的命令，必须复用同一套 CI 环境初始化路径，不能在不同 workflow 中各自维护一份隐式 bootstrap 脚本。

### 12.7 支持与客户闭环工作流

负责：issue intake、故障复现、优先级管理、RCA、试点反馈。

要求：

- 每个高优先级缺陷都必须形成经验沉淀与新增测试。

## 14. 里程碑 KPI 建议

以下 KPI 不是最终定值，但建议作为商业化推进的最小门槛。

### Alpha 目标

- 核心流程 CI 通过率 >= 95%。
- 无未归因崩溃类 blocker。
- 所有公共 CLI / Python 示例可运行。
- 关键 benchmark 均有基线结果。

### Beta 目标

- 核心流程 CI 通过率 >= 98%。
- 关键 schema 兼容性破坏为 0。
- 首批对照矩阵稳定运行 2 个迭代以上。
- 高优先级缺陷平均关闭时间 <= 14 天。

### GA 目标

- 核心流程 CI 通过率 >= 99%。
- 已承诺边界内无 P0 未解决缺陷。
- 发布流程全自动且可回滚。
- 外部试点用户能独立完成安装、运行、导入与报告解读。

## 15. 不允许跳过的事项

以下事项常常会被“先把功能做出来”挤掉，但如果目标是商业成熟，任何一项都不能跳过。

1. 版本化 schema 与兼容性测试。
2. benchmark 与外部对照验证。
3. PDK 验证与版本化。
4. 诊断包、结构化日志与可复现错误码。
5. 安装、升级、回滚流程。
6. 缺陷分级、RCA 与支持闭环。
7. 已知限制清单和明确的能力边界。

如果这些事情被长期推后，项目只会变成功能更多的研究原型，不会变成商业成熟产品。

## 18. 最终达成标准

只有当以下问题都能回答“是”，`rflux` 才能说自己达到了商业化成熟工具的标准：

- 是否能对外明确承诺支持边界，并在边界内稳定交付？
- 是否能对关键结果给出对照证据和误差范围？
- 是否能在版本升级时保护现有用户流程？
- 是否能在客户出问题时快速复现、定位并修复？
- 是否能稳定接入目标 PDK、library 和输入格式？
- 是否能通过发布、安装、升级、回滚和支持流程完成完整交付？

如果其中任一项答案仍然是否定的，那么项目仍然处于“高质量原型”阶段，而不是商业成熟阶段。

## 19. 执行建议

这份路线图的核心不是“做更多”，而是“在可证明的边界内做深”。

对 `rflux` 来说，最危险的路径不是做得慢，而是继续在 prototype 状态下横向加功能，最后形成一个功能看起来很多、但没有任何一块能被商业用户放心依赖的系统。

正确推进方式应该是：

- 先缩边界。
- 再做契约。
- 再做可信度。
- 再做性能与产品化。
- 最后再扩大承诺范围。

按这个顺序推进，才有可能真正达到商业化成熟工具的目标。

## 20. 对标项目未来 12 周执行表

本节把前文路线图进一步压缩为一个面向当前仓库状态的 12 周执行表。

目标不是在 12 周内“达到商业成熟”，而是用 12 周把当前最值钱的差距快速压缩到可验证、可继续扩展的状态。

本节默认对标以下几类项目：

- JoSIM：原生瞬态仿真与 deck 兼容基线。
- Yosys：前端、综合、SAT、等价检查与 CLI 契约基线。
- Quaigh：布尔优化深度与等价证据基线。
- OpenROAD / nextpnr：物理实现工程化基线。
- OpenSTA：时序相关性与 signoff 边界基线。

### 20.1 12 周总目标

12 周结束时，至少应达到以下状态：

- 有一条正式承诺的输入路径和一条受限支持路径。
- `rflux-sim` 与 JoSIM 的相关性对照进入可重复执行的门禁。
- PDK 接入不再只依赖手工约定，而是有最小 validate 入口。
- CLI / diagnostics / Python 对外错误语义继续收敛，未分类错误比例下降。
- P&R / timing 至少建立第一版 benchmark 和相关性基线。

### 20.2 周次安排

| 周次 | 主题 | 主要对标 | 主要交付 | 退出条件 |
|------|------|----------|----------|----------|
| 第 1 周 | 输入边界冻结 | Yosys / JoSIM | 冻结首批正式支持输入子集；补 `docs/support-matrix.md` 和输入支持表草案 | 仓库内已有“正式支持 / 受限支持 / 不支持”三档定义 |
| 第 2 周 | 输入契约回归 | Yosys | 为 IR JSON、PDK JSON、受限 deck 路径补 schema / 兼容回归；清点隐式 fallback | 关键输入路径都有成功 / 非法 / 不支持回归 |
| 第 3 周 | PDK validate 基线 | 商业产品 / OpenROAD | 增加最小 `pdk validate` 入口或等价 CLI；补 PDK onboarding 文档框架 | 自定义 PDK 不再只能靠人工试跑判断可用性 |
| 第 4 周 | 错误码收敛第一轮 | Yosys / 商业产品 | 继续压缩 `RFLOW-INTERNAL-001`；优先覆盖 `FLOW` / `VERIFY` / `SIM` 稳定边界 | 常见输入 / verify / sim 失败均输出稳定 `RFLOW-*` |
| 第 5 周 | JoSIM 相关性门禁起步 | JoSIM | 固化 compare 脚本输入、输出和阈值；挑 1 批关键 deck 进入手动或 nightly 门禁 | 至少一组 waveform / delay 对照结果可稳定复现 |
| 第 6 周 | deck 子集扩展 | JoSIM | 扩一小批最常见 deck 语法和 source/device 子集；每补一种语法都带对照用例 | 受限 deck 支持表显著扩大，且有回归证据 |
| 第 7 周 | 综合优化深化 | Quaigh / Yosys | 扩 XOR/MUX/DFFE 相关 pattern；补 fixture 和 SAT 等价证据 | 新增优化均有 fixture、QoR 变化和等价证明 |
| 第 8 周 | 验证能力扩展 | Yosys / Quaigh | 扩 `check-equivalence` 的接口/语义失败分类与回放能力；补更多 export/replay regression | verify 失败更可解释，可直接指导用户修复输入 |
| 第 9 周 | P&R benchmark 基线 | OpenROAD / nextpnr | 建立 placement / routing smoke benchmark、容量边界和失败模式记录 | 能回答“当前规模多大能跑、何种约束会失败” |
| 第 10 周 | timing correlation 起步 | OpenSTA | 选择一批关键 timing 场景建立外部或人工对照；补误差记录脚本 | 至少一版 timing correlation 结果可被审阅 |
| 第 11 周 | diagnostics 支持包升级 | 商业产品 | 在当前 bundle 之上补更完整配置快照、更多命令覆盖或 profile 开关 | 诊断包从“最小复现包”向“支持包”迈进一层 |
| 第 12 周 | Alpha 准入复盘 | 全部 | 输出 12 周收敛报告、剩余 gap、下一阶段目标和停止宣称边界 | 有一份可给外部试用用户的能力与限制说明 |

### 20.3 Top 10 任务映射到周次

| 高价值任务 | 主落点周次 | 说明 |
|------------|------------|------|
| 正式支持输入子集 | 第 1-2 周 | 先缩边界，再写契约和回归 |

| timing correlation | 第 10 周 | 先有基线，再谈 signoff 语义 |

| P&R 容量与 benchmark | 第 9 周 | 建容量边界和失败模式，比横向加功能更值钱 |
| 分发 / 安装链路 | 第 12 周后续阶段 | 可在 Alpha 准入复盘后进入下一轮 |

### 20.5 不该在这 12 周内做错的事

以下事情在 12 周内看起来很“有进展”，但实际上会稀释主线：

- 横向增加大量新命令，却不补输入契约和错误语义。
- 扩更多 deck 语法，却没有 JoSIM 数值对照阈值。
- 扩更多 P&R 选项，却没有 benchmark、容量边界和失败模式记录。
- 扩 Python facade 表面 API，却没有统一 CLI / Python / JSON 契约。
- 持续依赖 `minimal-sfq`，却不建立自定义 PDK validate 流程。

### 20.6 12 周后应如何判断是否继续同一主线

12 周结束后，若以下问题仍无法回答“是”，则不应贸然扩大支持范围：

- 输入路径是否已有一条正式承诺的稳定入口？
- JoSIM 对照是否已有至少一批可复现、可阈值化的证据？
- PDK 接入是否已有最小 validate 和文档？
- 常见失败是否已有稳定错误码与建议动作？
- 诊断包是否足以支撑外部试用期排障？

若上述问题大多已回答“是”，下一阶段再扩大：

- 更强 HDL frontend。
- 更广 SPICE / JoSIM 子集。
- 多 PDK 正式支持承诺。
- 更强 P&R / timing 相关性和 signoff 边界。
