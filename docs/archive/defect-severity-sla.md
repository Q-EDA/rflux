# rflux 缺陷分级与响应策略 v0

## 1. 目的

本文件定义 `rflux` 的缺陷分级、优先级和响应目标，用于内部治理、共研支持和后续商业化支持体系。

当前为 v0，先定义规则，再逐步接入 issue 管理与发布流程。

## 2. 严重级别

### P0

定义：

- 核心命令不可用。
- 输出结果被确认严重错误，且会直接误导设计决策。
- 发布产物不可安装或无法启动。
- 崩溃、死锁或数据损坏影响主要支持路径。

处理要求：

- 立即响应。
- 中止相关发布或试点扩展。
- 必须有临时缓解方案或回滚方案。

### P1

定义：

- 关键工作流在常见输入上失败。
- 兼容性破坏影响现有用户流程。
- 关键 benchmark 出现未解释重大退化。

处理要求：

- 进入当前迭代最高优先级队列。

### P2

定义：

- 非关键工作流失败。
- 结果正确但可观测性、错误信息或易用性明显不足。
- 局部性能问题影响效率但不阻断交付。

### P3

定义：

- 文档错误。
- 边缘功能缺陷。
- 小型改进建议。

## 3. 响应目标

当前建议目标如下：

| 严重级别 | 首次响应目标 | 修复或处置目标 |
|----------|--------------|----------------|
| P0 | 1 个工作日内 | 3 个工作日内给出处置方案 |
| P1 | 2 个工作日内 | 10 个工作日内进入修复或回滚计划 |
| P2 | 5 个工作日内 | 进入版本排期 |
| P3 | 下次常规 triage | 视资源安排 |

注：当前仓库尚未建立正式客服/值班体系，因此该目标应视为内部执行目标，而非外部合同 SLA。

## 4. 必须记录的信息

所有 P0 / P1 缺陷必须记录：

- 影响范围。
- 首次发现版本。
- 可复现步骤。
- 是否有 workaround。
- 是否需要回滚。
- 是否影响已发布结果可信度。

## 5. RCA 触发条件

以下情况必须做根因分析：

- 任意 P0。
- 连续两次出现同类 P1。
- 因默认行为或 silent fallback 导致的问题。
- 因 schema / 兼容性破坏导致的问题。

## 6. 修复准入条件

任何 P0 / P1 修复都必须满足：

- 有最小复现用例。
- 有回归测试。
- 更新已知限制或发布说明。
- 说明是否影响既有结果。

## 7. 当前执行建议

短期内建议把所有 issue 增加以下标签：

- `severity:P0`
- `severity:P1`
- `severity:P2`
- `severity:P3`
- `area:cli`
- `area:python`
- `area:synth`
- `area:place-route`
- `area:timing`
- `area:sim`
- `area:pdk`

这样后续才能形成支持与发布统计。
