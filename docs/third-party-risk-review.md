# rflux 第三方组件风险审查清单 v0

## 1. 目的

本文件定义 `rflux` 在引入、升级或继续携带第三方组件时的最小风险审查要求。

它服务于商业化路线图中的“第三方组件风险审查”，目标是让依赖治理从“知道用了什么”推进到“知道为什么可以用、风险在哪里、由谁批准继续使用”。

## 2. 适用范围

本清单适用于以下对象：

- Rust crate 依赖
- Python package 依赖
- 外部命令工具，例如 `josim`
- 将来纳入构建、测试、发布流程的第三方二进制或服务

## 3. 审查触发条件

以下情况至少触发一次审查：

1. 新增第三方组件。
2. 升级现有第三方组件主版本或关键次版本。
3. 将实验性依赖提升为默认或正式支持路径。
4. 把外部工具执行路径暴露到 CLI 或 Python 公共接口。
5. 发现新的许可证、供应链或安全告警。

## 4. 最小审查项目

每个第三方组件至少要记录以下结论：

| 项目 | 要求 |
|------|------|
| 组件标识 | 名称、版本、来源 |
| 引入目的 | 为什么必须引入，替代方案是什么 |
| 执行面 | 仅构建期、仅测试期、运行期，还是对外接口路径 |
| 许可证 | 许可证类型、是否与当前发布策略兼容 |
| 安全面 | 是否有已知告警、是否需要例外审批 |
| 支持面 | 是否进入正式支持矩阵，还是仅内部使用 |
| 负责人 | 谁负责后续升级、漏洞跟踪和例外处理 |

## 5. 风险分级

建议先按以下粗粒度分级执行：

| 级别 | 含义 | 当前处理要求 |
|------|------|--------------|
| R1 | 构建/测试期工具，不进入运行时交付 | 记录来源与许可证 |
| R2 | 运行时依赖，但不直接暴露执行面 | 记录来源、许可证、安全状态、负责人 |
| R3 | 直接形成外部执行面或公共接口依赖 | 需要显式边界文档、测试和发布审查 |

当前仓库中的 `josim` 外部调用路径，应至少按 `R3` 处理。

## 6. 当前执行建议

当前阶段建议把审查结果至少沉淀为以下三类工件：

- 自动导出的依赖清单
- 自动或半自动导出的许可证清单
- 本清单驱动的人工审查记录与例外台账

如果某项自动化暂时无法做到 authoritative，必须在审查记录中明确标注“人工确认”或“待补自动化”，不能默认为已闭环。

当前执行台账见 `docs/third-party-risk-register.md`。
例外审批和人工补审可使用 `docs/third-party-exception-template.md`。
当前已落首条例外记录：`docs/third-party-exception-tpr-001.md`。

## 7. 退出条件

只有当以下条件同时满足时，才可以认为第三方组件风险审查进入可交付状态：

- Rust 与 Python 依赖都能产出机器可读清单。
- 许可证清单具备可审查来源。
- R3 级组件均有边界文档和回归测试。
- 例外项目已经登记并具备关闭条件。
- 发布前检查明确包含第三方组件风险审查。
