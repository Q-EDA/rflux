# rflux 首个商业化范围说明书 v0

## 1. 目的

本文件定义 `rflux` 首个商业化版本的建议产品边界。目标不是覆盖尽可能多的能力，而是在一个可证明、可维护、可支持的范围内做到稳定交付。

## 2. 推荐首发范围

首个对外可承诺版本应定位为：

- 面向 SFQ 电路的受限输入、受限工艺、受限规模设计分析与验证工具链。
- 以 IR JSON 为主输入路径。
- 以 CLI + Python facade 为主要交付面。
- 以综合、原型级物理实现、确定性时序、结构验证、受限仿真和等价检查为首发能力。

## 3. 首发承诺能力

### 3.1 输入与建模

- `rflux-ir` JSON 作为正式输入基线。
- Rust `rflux-hdl` builder DSL 作为内部建模与示例输入路径。
- 受限 SPICE / JoSIM deck 作为仿真专用输入路径。

当前收口说明：

- 对外试用、benchmark 和支持工单默认应优先收敛到 `rflux-ir` JSON 复现。
- Rust `rflux-hdl` builder DSL 目前只应视为内部建模与测试辅助入口，不应被表述为通用 HDL frontend。
- 受限 SPICE / JoSIM deck 当前仅应被表述为仿真专用、受限支持输入，不应外推为通用 SPICE 兼容承诺。

### 3.2 核心流程

- `compile-netlist`
- `compile-layout`
- `analyze-timing`
- `verify-layout`
- `solve-dimacs`
- `check-equivalence`
- `simulate-file` / `simulate-text` 的受限子集

### 3.3 交付面

- Rust CLI
- Python facade
- 版本化文档
- 样例与 benchmark 套件

## 4. 首发不承诺能力

以下能力不应进入首个商业化承诺范围：

- 通用 Verilog frontend。
- signoff 级 P&R。
- signoff 级 STA / SSTA。
- 通用 SPICE 兼容仿真器。
- 完整 JoSIM 语义兼容。
- 多 PDK 商业化长期支持。
- 未版本化的 schema 演进。

补充说明：

- 在输入层面，这意味着首发版本不应对“任意 HDL 输入可导入”或“任意 SPICE deck 可仿真”作出承诺。

## 5. 目标用户

优先面向以下用户：

- SFQ 方向研发团队。
- 需要脚本化、批处理和实验可重复性的内部 CAD 团队。
- 能接受受限输入和已知限制、但需要比原型更稳定工具链的共研用户。

不优先面向：

- 希望把 `rflux` 当作通用 CMOS EDA 替代品的用户。
- 不具备共研和对照验证能力的纯黑盒终端用户。

## 6. 支持规模建议

首发商业版本必须先声明目标规模，而不是隐含“尽可能大”。建议通过 benchmark 与真实样例在后续版本中填充以下表格：

| 场景 | 建议状态 |
|------|----------|
| 小规模研究样例 | 首发必须支持 |
| 中等规模验证样例 | 首发建议支持 |
| 大规模签核级样例 | 首发不承诺 |

## 7. 首发成功标准

只有当以下条件同时满足，才说明首个商业范围定义正确：

- 支持矩阵可明确列出正式支持项与受限支持项。
- 对关键结果已有外部对照计划。
- 共研用户可在无作者陪跑的情况下完成安装、导入、运行和结果读取。
- 缺陷、升级、兼容性和已知限制均有正式文档。

## 8. 变更规则

本文件中“首发承诺能力”的任何扩展都必须满足：

- 已有实现。
- 已有回归测试。
- 已更新支持矩阵。
- 已知限制已同步收缩。
- 支持与发布团队确认可以维护。
